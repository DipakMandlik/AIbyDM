# Good First Issues

Good first issues should help new contributors succeed quickly while improving the platform in visible ways.

## Good First Issue Criteria

- The scope is small and reviewable.
- The expected outcome is clear.
- The task does not require private context.
- The contributor can validate the change locally.
- The issue includes links to the files or docs likely to be touched.

## Recommended Issue Types

| Type | Examples |
| --- | --- |
| Docs | Clarify setup, improve README sections, update architecture notes. |
| Content | Add lesson examples, improve summaries, fix metadata, add exam explanations. |
| Tools | Add or improve one AI tool entry with use cases and alternatives. |
| UX polish | Improve copy, accessibility, mobile spacing, or empty states on one page. |
| Community | Improve templates, labels, contributor recognition, or release notes. |

## Labels

Use these labels for contributor-friendly tasks:

- `good first issue`
- `help wanted`
- `type:docs`
- `type:content`
- `type:tool`
- `status:needs-triage`

## Issue Template

Every good first issue should include:

1. Problem or opportunity.
2. Suggested files to inspect.
3. Acceptance criteria.
4. Validation command or visual check.
5. Maintainer contact path through GitHub Discussions.
